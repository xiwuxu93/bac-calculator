# BAC Calculator 性能、体验与无障碍检查清单

> 目的：为后续开发与上线前验证提供一份集中清单，确保 BAC 计算器在移动端和桌面端都有良好体验，并符合基本可访问性要求。

---

## 1. 布局与首屏体验

- [ ] 首页在常见手机尺寸下，Hero + 工具上半部分可以在首屏看到  
- [ ] 工具区域使用简洁卡片布局，重要控制（性别、体重、时间、饮品列表、Estimate 按钮）清晰可见  
- [ ] 主要按钮与表单控件不会被固定导航或广告遮挡  
- [ ] 所有文本在移动端至少使用 16px 等效字号

---

## 2. 交互与可用性

- [ ] 所有输入字段有明确标签（label）与示例占位符（如适用）  
- [ ] 数字输入在移动端拉起数字键盘（使用合适的 input 类型）  
- [ ] “Estimate BAC” 按钮在表单底部固定位置和视觉上突出  
- [ ] 校验错误时提供清晰文本提示，并高亮对应字段  
- [ ] 重置/清空功能明显，但不会误触（可加确认或放在次级位置）  
- [ ] 在结果区域，风险等级与法律信息用清晰标题（例如 “Estimated BAC”, “Legal Risk”, “Safety Recommendation”）区分

---

## 3. 可访问性（Accessibility）

- [ ] 页面标题（`<h1>`）唯一且描述清晰  
- [ ] 所有交互控件（按钮、链接、radio、select）支持键盘导航  
- [ ] 焦点状态可见（有明显 focus 样式）  
- [ ] 对于彩色风险标签（如红/黄/绿），同时提供文字说明，不仅依赖颜色区分  
- [ ] 所有非装饰性图像（如 og 图、图标）具备合适的 `alt` 文本  
- [ ] 表单校验错误通过 ARIA 属性与字段关联（后续实现时需注意）

---

## 4. 性能目标

- [ ] 移动端首屏加载（LCP）目标 < 2.5 秒（PageSpeed Insights）  
- [ ] 初始 JS 体积控制在合理范围内（避免引入大型未使用库）  
- [ ] 工具主要逻辑在客户端执行，无额外网络请求导致延迟  
- [ ] 延迟加载非关键内容（如底部说明图片或不必要脚本）  
- [ ] 图片（包括未来的插图）使用现代格式（如 WebP 或 SVG），并限制体积

---

## 5. 技术实现注意事项

- [ ] 使用 Next.js 内置的优化（如动态导入、`next/image` 等）时，保持实现简单可维护  
- [ ] 避免在客户端组件中做不必要的繁重计算（复杂逻辑可拆到 utility 函数并适当 memo）  
- [ ] 对于 localStorage 访问，使用 `useEffect` 等在客户端安全调用，避免 SSR 报错  
- [ ] 所有依赖（第三方包）在引入前评估必要性，避免为简单功能引入大型库

---

## 6. 测试计划（上线前）

- [ ] 使用 Lighthouse 对首页进行一次完整审查（Performance / Accessibility / Best Practices / SEO）  
- [ ] 在 Chrome、Safari、Firefox、Edge 最新版本中手动测试表单与结果显示  
- [ ] 在 iOS 和 Android 上实际设备测试（至少各 1–2 种分辨率）  
- [ ] 使用无痕模式访问，验证在无 localStorage 预设时表现正常  
- [ ] 如有启用 GA 或广告代码，确认不会明显拖慢首屏加载时间

---

本清单将在开发和上线前多次使用：  
- 开发过程中作为“设计约束”；  
- 实现完成后作为自测基准；  
- 上线后可继续作为迭代与重构的参考标准。
